# LOCAL HOST OPERATING PROTOCOL v1
> **Purpose:** Define how human hosts operate within Muonnoi Life Quest OS
> **Scope:** All Local Hosts in Đường Muôn Nơi pilot (Đà Lạt, 2026-08)
> **Status:** LOCKED — DO NOT ALTER WITHOUT FOUNDER APPROVAL
> **Locked By:** Founder, 2026-06-05

---

## 1. HOST DEFINITION

A **Local Host** is:
- An independent individual or small group
- Verified KYC Tier 3
- Approved for a specific quest system and city
- Not an employee of Muonnoi.org
- Operating under independent contractor terms

A Local Host is NOT:
- A tour operator
- An employee of Muonnoi
- A travel agency representative
- Guaranteed income or employment

---

## 2. ONBOARDING FLOW

### Step 1: Application
- Host fills `/host/register` form
- Provides: name, email, phone, city, quest types they can lead, bio (200 chars max)
- Submits: 2 photos (portrait, activity context photo)

### Step 2: KYC Tier 2 (Online)
- Email + phone OTP
- CCCD/CMND front + back photo
- Selfie with liveness check (blink detection)
- Automatic approval within 24h if documents clear

### Step 3: KYC Tier 3 (Offline — REQUIRED for Local Host)
- **Interviewer:** Founder or appointed Ops Lead
- **Location:** In-person in city (Đà Lạt) or video call if Founder remote
- **Duration:** 60-90 minutes
- **Components:**
  1. Background check conversation (15 min)
  2. Quest-specific knowledge test (15 min)
  3. Safety scenario role-play (20 min)
  4. Communication style assessment (15 min)
  5. SOP briefing + Q&A (15 min)

### Step 4: Training
- **Duration:** 4 hours minimum
- **Format:** In-person workshop or video + quiz
- **Modules:**
  1. Muonnoi values + anti-addiction principles (30 min)
  2. Quest content walkthrough (60 min)
  3. Proof acceptance criteria (30 min)
  4. Safety protocols + emergency procedures (60 min)
  5. App usage: accept/reject proof, communicate with user (30 min)
  6. Review etiquette + no-show handling (30 min)

### Step 5: Reference Check
- **Count:** 2 references minimum
- **Type:** 1 personal + 1 professional (previous host/guide/employer)
- **Method:** Phone call or email, not automated
- **Questions:**
  - "Would you recommend this person to care for your family member on a trip?"
  - "Have you seen them handle a difficult situation? How?"
  - "Any concerns about safety, honesty, or reliability?"

### Step 6: Activation
- Host signs `LOCAL_HOST_AGREEMENT_v1.pdf`
- SOP acknowledgement signed
- Assigned 1-3 quests from available pool
- Added to host Slack/communication channel
- Given host dashboard access

## 3. COMMUNICATION RULES

### Host ↔ User
| Channel | Allowed | Not Allowed |
|---------|---------|-------------|
| In-app messaging | ✅ Yes — primary channel | — |
| Phone (after booking) | ✅ Yes — for logistics only | ❌ No for booking, payment, cancellation |
| WhatsApp/Zalo (personal) | ⚠️ Only if user initiates and asks | ❌ Host cannot proactively share personal contact before quest |
| Social media friend request | ❌ No | — |
| Meeting outside quest context | ❌ No | — |

### Response Time
- User message → host replies within **4 hours** during business hours (08:00-20:00)
- Urgent (safety, lost, medical) → **within 30 minutes**
- No response for 24h → auto-escalated to Ops Lead

### Tone
- Friendly but professional
- No sales pitch, no upselling outside quest scope
- Answer questions honestly; if don't know, say "I don't know, let me check"
- No political, religious, or discriminatory content

---

## 4. USER CHECK-IN

### Before Quest
- Host confirms with user 24h before via app message
- Reminds: meet location, time, weather, required gear
- Answers any last-minute questions

### At Meet Point
- Host arrives **15 minutes early**
- Verifies user identity (name match app profile, optional photo check)
- Briefs: quest plan, safety notes, proof requirements, duration
- Checks: user health (any allergies, injuries, limitations?)

### During Quest
- Host leads at appropriate pace
- Explains, does not lecture
- Encourages questions and pauses
- Monitors group energy and adjusts
- Takes photos of user (with permission) for their proof if needed
- Does NOT use phone for personal matters during quest

### After Quest
- Host thanks user
- Reminds user to submit proof within 24h
- Answers any follow-up questions
- Does NOT ask for tip, review, or social media follow

## 5. PROOF ACCEPTANCE RULES

### Host Authority
- Host is the **sole reviewer** for their own quest proofs
- Host can: **accept**, **reject**, or **request resubmission**
- Host CANNOT modify user proof
- Host CANNOT delete user account

### Accept Criteria
Proof is ACCEPTED if:
- [ ] Photo/text submitted within 48h of quest completion
- [ ] Photo is authentic (not screenshot, not stock image)
- [ ] GPS (if required) is within approved radius
- [ ] Text is original (not copy-paste from internet)
- [ ] Content is appropriate (no nudity, hate speech, violence)

### Reject Criteria
Proof is REJECTED if:
- [ ] Missing required proof type
- [ ] Photo is clearly fake or unrelated
- [ ] GPS is outside radius by >50%
- [ ] Text is blank, gibberish, or copy-paste
- [ ] Submitted >7 days after quest (unless prior arrangement)

### Reject Response
Host MUST provide reason when rejecting:
- "Photo doesn't show you at the location. Please resubmit with GPS on."
- "Text appears copied from internet. Please write your own reflection."
- "Missing GPS proof. Please submit with location enabled."

### Escalation
If user appeals rejection:
1. Host re-reviews with Ops Lead
2. Ops Lead makes final call
3. Both host and user are notified
4. No penalty to host for honest mistake

## 6. NO-SHOW HANDLING

### User No-Show
- User doesn't arrive within 15 minutes of meet time
- Host sends 1 app message + 1 phone call (if phone available)
- Wait 30 minutes total
- If no contact: mark "user no-show" in dashboard, quest cancelled
- User receives "no-show" note (no penalty for first time, warning for repeat)

### Host No-Show
- If host cannot make it: notify Ops Lead **minimum 4 hours before**
- Ops Lead finds replacement or cancels + notifies user
- If host no-show without notice: **suspension pending investigation**
- 2 confirmed no-shows → **deactivation from host pool**

## 7. SAFETY ESCALATION

### Low Severity (Level 1)
- Minor injury (scratch, blister)
- User feels unwell (headache, nausea)
- Weather change affecting quest

**Action:** Host handles on-site. First aid kit. Adjust quest if needed. Log in app after.

### Medium Severity (Level 2)
- Moderate injury (sprain, cut needing stitches)
- User lost/separated from group
- Host-user conflict (argument, discomfort)

**Action:**
1. Host secures scene
2. Calls Muonnoi safety hotline
3. Ops Lead coordinates support
4. If medical: call 115 (VN emergency) or nearest clinic
5. Log incident within 2 hours

### High Severity (Level 3)
- Serious injury requiring hospital
- Host/user harassment or assault
- Fraud or theft
- Legal involvement (police)

**Action:**
1. Host ensures immediate safety (remove from danger)
2. Calls 115 + police 113 if crime
3. Calls Muonnoi safety hotline immediately
4. Founder notified within 30 minutes
5. Full incident report within 4 hours
6. **Pilot may be paused pending investigation**

### Emergency (Level 4)
- Life-threatening injury
- Natural disaster
- Violence with weapon
- Missing person >2 hours

**Action:**
1. Call 115/113 immediately
2. Founder notified immediately (phone call, not message)
3. **Pilot STOPs** until Founder clears restart
4. External investigation if needed
5. Media/inquiry → only Founder speaks, not host

## 8. INCIDENT REPORTING

Every Level 2+ incident requires:

```
INCIDENT REPORT — [Date] — [Level]
1. Time and location
2. People involved (host, user, witnesses)
3. What happened (facts only, no interpretation)
4. Immediate actions taken
5. Current status (resolved/ongoing)
6. Follow-up needed
7. Prevention suggestion
```

Report filed in `qa/incidents/YYYY-MM-DD-{brief-description}.md`

---

## 9. REVIEW ETIQUETTE

### Host Reviews User
- Honest but kind
- Focus on behavior, not personality
- Example good: "User arrived prepared with proper shoes. Asked thoughtful questions."
- Example bad: "User was annoying and lazy."

### User Reviews Host
- Host cannot see review until they submit their own
- Host cannot modify or delete user review
- Host can respond once (professional tone)

### Review Abuse
- Host asking for positive review → warning
- Host threatening negative consequences for bad review → deactivation
- User fake review (never attended) → user account suspension

---

## 10. ANTI-ABUSE

### Host Abuse (automatic deactivation)
- Physical or sexual harassment of user
- Theft or fraud
- Drug/alcohol use during quest
- Selling goods/services outside quest scope without consent
- Sharing user personal data
- Ghost quest (advertised but not real)

### User Abuse (host reports to Ops Lead)
- Harassment of host
- Property damage
- Refusing to follow safety instructions
- Illegal activity during quest

## 11. DATA RETENTION

| Data Type | Retention | After Retention |
|-----------|-----------|-----------------|
| Host application | 2 years | Anonymize |
| KYC documents (photos) | 2 years after host deactivation | Secure delete |
| Quest photos (host took of user) | 1 year | User consent required to keep |
| Incident reports | 5 years | Archive |
| Communication logs | 1 year | Delete |

Host can request deletion of personal data anytime. 30-day processing.
